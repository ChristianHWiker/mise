Implementation Spec · v0.1 draft

Recipe Calculator

A mobile-first recipe manager that scales recipes the way cooks actually think — fractional units, smart rounding, and reverse scaling from a single limiting ingredient.

Owner  chwi@pelias.no Status  Drafting Target  Play Store, Android first

Problem

What existing apps miss.

The Play Store has dozens of recipe apps. Almost all of them treat scaling as a single operation: multiply every quantity by a target serving count. That's the easy half of how cooks actually scale recipes.

The other half — the half no mainstream app does well — is:

  • Reverse scaling. "I have 350 g of chicken. Scale the rest of the recipe to match."
  • Fractional output. Cooking lives in 3/4 cup and 1½ tsp, not 0.75 and 1.5. Decimal output looks wrong and is harder to measure.
  • Ingredient-aware unit conversion. 1 cup of flour ≠ 1 cup of sugar by weight. Most converters quietly get this wrong.
  • Smart rounding. Scaling 3 eggs by 1.5× should suggest 4 eggs (adjusted), not 4.5 eggs.

These are unglamorous but obvious gaps once you've cooked from a real recipe. They're the spine of the product.

Approach

The stack and the load-bearing decisions.

Stack

  • React Native (Expo) with TypeScript for the cross-platform shell and UI.
  • One Kotlin native module wrapping Android ML Kit text recognition for the camera/OCR path.
  • SQLite for local persistence (via expo-sqlite; revisit WatermelonDB if reactivity becomes painful).
  • Rational quantities — numerator/denominator integers, never floats — so 3/4 + 1/8 stays exact.
  • Ingredient density table — curated lookup (~150 common ingredients) for volume↔mass conversion.

Why this, not that

RN over Flutter

Stronger job-market signal in EU/Norway. Flutter has nicer DX but RN is more frequently named in listings.

Kotlin native module, not pure JS

ML Kit is a native API. Wrapping it as a JS-bridged Kotlin module also showcases native Android skill — a deliberate portfolio signal.

Local-only v1

Sync is infrastructure. Ship the core UX first; add accounts when there's evidence anyone wants their recipes on two devices.

OCR with confirmation screen

Bounded scope. Recognizing text is solved. Parsing OCR output into clean structured ingredients is not — confirmation keeps quality honest.

Rational quantities, not floats

Fractional display "just works" with no rounding loss. Floats force display-time reconstruction and accumulate error across operations.

SQLite, not a JSON file

Search, indexing, and migration story all need a real DB once recipe count grows. Cheap to set up via Expo.

Data model

The minimum viable shape. Lossless quantities are the load-bearing choice.

data-model.tsTypeScript
// Rational quantity — never lose 3/4 to a float.
type Quantity = { num: number; den: number };

type UnitSystem = "metric" | "us";
type UnitKind   = "volume" | "mass" | "count" | "length";

interface Unit {
  code: string;            // "cup", "g", "tsp", "ea"
  kind: UnitKind;
  system: UnitSystem;
  baseFactor: number;      // convert to canonical (ml or g)
}

interface Ingredient {
  id: string;
  name: string;            // "all-purpose flour"
  quantity: Quantity;
  unit: string;            // Unit.code
  modifier?: string;       // "sifted", "room temp"
  optional?: boolean;
  order: number;
}

interface Recipe {
  id: string;
  title: string;
  servings: number;
  ingredients: Ingredient[];
  steps: string[];
  tags: string[];
  photoUri?: string;
  sourceUrl?: string;
  defaultUnitSystem: UnitSystem;
  createdAt: number;
  updatedAt: number;
}

// Curated table — gramsPerCup for ingredient-aware conversion.
interface Density {
  ingredientKey: string; // normalized: "flour-ap", "sugar-granulated"
  gramsPerCup: number;
  aliases: string[];     // "plain flour", "AP flour"
}
Load-bearing decision

Storing quantities as {num, den} instead of number is the choice that makes fractional display, scaling, and unit conversion all stay clean downstream. Every other shortcut here you can change later. This one you can't.

Phased roadmap

Each phase is independently shippable. Estimates assume part-time pace.

00Foundations

Expo TS project skeleton, navigation, theme tokens, SQLite migrations, rational-number utility with tests.

~1 week
Done when: app builds and runs on emulator; a dev-only screen can insert a recipe and read it back from SQLite.

01Recipe CRUD

List, detail, create/edit screens. Structured ingredient editor (qty, unit, name, modifier). Tags, search, favorites.

~1–2 weeks
Done when: user can manually enter a real recipe, save it, find it via search or tags, edit it, delete it.

02Scaling engine

Forward scaling by servings. Reverse scaling by limiting ingredient. Fractional display. Smart rounding (counts → integer, volumes/masses → nearest "useful" fraction).

~1 week
Done when: a 4-serving recipe scaled to 6 displays sensible fractions; reverse-scaling from any ingredient produces correct totals; whole-unit ingredients round.

03Unit system

EU/NA toggle in settings & per-recipe override. Density table for ~150 common ingredients. Ingredient-aware conversion with fallback to volumetric-only when ingredient is unknown.

~1 week
Done when: toggling unit system on a recipe changes display; flour at 1 cup → 120 g, sugar at 1 cup → 200 g; unknown ingredients fall back gracefully without crashing.

04OCR (camera capture)

Kotlin native module wrapping ML Kit text recognition. JS bridge. Ingredient line parser (regex + heuristics). Confirmation screen with editable parsed rows.

~2 weeks
Done when: user can photograph a printed ingredient list, see parsed rows on a confirmation screen, fix any mis-parses inline, and save as a new recipe.

05Polish & launch

Onboarding flow. Settings (default units, default servings). App icon, splash screen, store assets. Signed AAB. Privacy policy. Play Store internal testing track.

~1 week
Done when: the AAB passes Play Store internal review and is installable from a tester invite.

Total estimate: ~7–8 weeks part-time. Phases 0–3 ship a credible app on their own; phase 4 is where the differentiator lands; phase 5 is launch hygiene.

Open questions

Risks

What can sink this and what to do about it.

RiskSeverityMitigation
Ingredient parser quality on real-world photos is too low — users abandon the OCR flow. Warning Confirmation screen sets expectations. Track parse-success metric internally. Iterate the regex set against a captured corpus of real ingredient lists.
Density table coverage gaps produce visibly wrong unit conversions. Warning Fall back to volume-only conversion for unknown ingredients. Show a small "approximate" indicator. Add an "improve this" tap target that logs the miss locally.
Smart rounding rules surprise users on edge cases ("why did my 1.5 eggs become 2?"). Info Always show an adjusted marker beside rounded values. Tap reveals the exact unrounded number.
Native module bridge work eats more time than planned. Info De-risk in phase 00 — get a "hello world" Kotlin module returning a string into JS before phase 01 starts. If it's painful, the OCR phase's estimate doubles, not the rest of the project.
Play Store rejection on first submission (icon, privacy policy, permissions justification). Info Submit to internal testing track early in phase 05, not at the end.
Scope creep — meal planning, sharing, sync get added to v1. Critical The non-goals section is the answer. Re-read it when tempted.

Portfolio angle

What this project demonstrates if you point a recruiter at the repo.

  • Cross-platform mobile. Modern React Native + Expo + TypeScript, the stack most EU/Norwegian listings actually ask for.
  • Native interop. A real Kotlin native module bridged into JS — shows you can drop into platform code when the JS layer can't reach.
  • On-device ML. ML Kit text recognition running locally, no server round-trip, no privacy story to defend.
  • Domain modeling. Rational quantities, ingredient-aware unit conversion, smart rounding — non-trivial logic that's easy to evaluate.
  • Shipped, not just coded. A live Play Store listing (even internal testing) ends every "have you shipped?" conversation in one sentence.
