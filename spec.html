Implementation Spec · v0.2 draft

Recipe Calculator

A mobile-first recipe manager that scales recipes the way cooks actually think — fractional units, smart rounding, and reverse scaling from a single limiting ingredient.

Owner  chwi@pelias.no Status  Drafting Target  Play Store · native Android

Problem

What existing apps miss.

The Play Store has dozens of recipe apps. Almost all of them treat scaling as a single operation: multiply every quantity by a target serving count. That's the easy half of how cooks actually scale recipes.

The other half — the half no mainstream app does well — is:

  • Reverse scaling. "I have 350 g of chicken. Scale the rest of the recipe to match."
  • Fractional output. Cooking lives in 3/4 cup and 1½ tsp, not 0.75 and 1.5. Decimal output looks wrong and is harder to measure.
  • Ingredient-aware unit conversion. 1 cup of flour ≠ 1 cup of sugar by weight. Most converters quietly get this wrong.
  • Smart rounding. Scaling 3 eggs by 1.5× should suggest 4 eggs (adjusted), not 4.5 eggs.

These are unglamorous but obvious gaps once you've cooked from a real recipe. They're the spine of the product.

Approach

The stack and the load-bearing decisions.

Stack

  • Native Android — Kotlin + Jetpack Compose (Material 3). Single-platform, no XML layouts.
  • MVVM + Coroutines/Flow for state; Hilt for dependency injection; Navigation Compose (single-activity).
  • Room for local persistence (SQLite underneath) with versioned, forward-only migrations.
  • ML Kit Text Recognition (on-device) + CameraX for the camera/OCR path — the native API used directly, no JS bridge.
  • AndroidX Biometric, Jetpack Security/Keystore, Play Integrity for app lock, encrypted storage, and integrity/anti-tampering.
  • Rational quantities — numerator/denominator integers, never floats — so 3/4 + 1/8 stays exact.
  • Ingredient density table — curated lookup (~150 common ingredients) for volume↔mass conversion.

Why this, not that

Native Kotlin over React Native

The target listing accepts either, but every bonus criterion — camera, biometrics, secure storage, integrity, on-device ML — is a native API. Native depth is the differentiated signal.

Jetpack Compose over XML views

Declarative UI is the modern Android standard and what current listings expect. Shows present-day skills, not legacy findViewById habits.

ML Kit direct, not a JS bridge

OCR was the riskiest part of the old plan. Going native turns an RN↔Kotlin bridge into a plain on-device API call — less risk, no round-trip, no privacy story to defend.

A dedicated security phase

Biometric app lock, Keystore-backed encryption, and Play Integrity attestation map straight onto the listing's "secure storage, integrity, anti-tampering" bullet.

Local-only v1

Sync is infrastructure. Ship the core UX first; add accounts when there's evidence anyone wants their recipes on two devices.

Rational quantities, not floats

Fractional display "just works" with no rounding loss. Floats force display-time reconstruction and accumulate error across operations.

Data model

The minimum viable shape. Lossless quantities are the load-bearing choice.

data/model/Recipe.ktKotlin
// Rational quantity — never lose 3/4 to a float.
data class Quantity(val num: Int, val den: Int)

enum class UnitSystem { METRIC, US }
enum class UnitKind   { VOLUME, MASS, COUNT, LENGTH }

data class MeasureUnit(
  val code: String,        // "cup", "g", "tsp", "ea"
  val kind: UnitKind,
  val system: UnitSystem,
  val baseFactor: Double,  // convert to canonical (ml or g)
)

@Entity(tableName = "ingredients")
data class Ingredient(
  @PrimaryKey val id: String,
  val name: String,        // "all-purpose flour"
  @Embedded val quantity: Quantity,
  val unit: String,        // MeasureUnit.code
  val modifier: String?,   // "sifted", "room temp"
  val optional: Boolean = false,
  val position: Int,
)

@Entity(tableName = "recipes")
data class Recipe(
  @PrimaryKey val id: String,
  val title: String,
  val servings: Int,
  val photoUri: String?,
  val sourceUrl: String?,
  val defaultUnitSystem: UnitSystem,
  val createdAt: Long,
  val updatedAt: Long,
)
// Ingredients, steps, and tags hang off Recipe via Room @Relation.

// Curated table — gramsPerCup for ingredient-aware conversion.
data class Density(
  val ingredientKey: String, // "flour-ap", "sugar-granulated"
  val gramsPerCup: Double,
  val aliases: List<String>,  // "plain flour", "AP flour"
)
Load-bearing decision

Storing quantities as {num, den} instead of number is the choice that makes fractional display, scaling, and unit conversion all stay clean downstream. Every other shortcut here you can change later. This one you can't.

Phased roadmap

Each phase is independently shippable. Estimates assume part-time pace.

00Foundations

Android Studio Kotlin project, Gradle Kotlin DSL + version catalog, Compose + Material 3 theme, Navigation Compose shell, Hilt, Room migrations, rational-number library with JUnit tests.

~1 week
Done when: app builds and runs on emulator; a dev-only screen can insert a recipe and read it back from Room.

01Recipe CRUD

List, detail, create/edit Compose screens. Structured ingredient editor (qty, unit, name, modifier). Tags, search, favorites. MVVM + Flow wiring.

~1–2 weeks
Done when: user can manually enter a real recipe, save it, find it via search or tags, edit it, delete it.

02Scaling engine

Forward scaling by servings. Reverse scaling by limiting ingredient. Fractional display. Smart rounding (counts → integer, volumes/masses → nearest "useful" fraction). Pure Kotlin, unit-tested.

~1 week
Done when: a 4-serving recipe scaled to 6 displays sensible fractions; reverse-scaling from any ingredient produces correct totals; whole-unit ingredients round.

03Unit system

EU/NA toggle in settings & per-recipe override. Density table for ~150 common ingredients. Ingredient-aware conversion with fallback to volumetric-only when ingredient is unknown.

~1 week
Done when: toggling unit system on a recipe changes display; flour at 1 cup → 120 g, sugar at 1 cup → 200 g; unknown ingredients fall back gracefully without crashing.

04OCR (camera capture)

CameraX capture flow. ML Kit on-device text recognition (called directly, no bridge). Ingredient line parser (regex + heuristics). Confirmation screen with editable parsed rows.

~2 weeks
Done when: user can photograph a printed ingredient list, see parsed rows on a confirmation screen, fix any mis-parses inline, and save as a new recipe.

05Security & hardening

Biometric app lock (AndroidX Biometric). Keystore-backed encryption for sensitive prefs / lockable recipes (Jetpack Security). Play Integrity attestation on launch. Basic anti-tampering signals; scoped backup rules.

~1–1.5 weeks
Done when: the app can be locked behind biometric auth; sensitive data is encrypted at rest; a failed Integrity verdict is detected and handled gracefully.

06Polish & launch

Onboarding flow. Settings (default units, default servings). App icon, splash screen, store assets. Signed AAB. Privacy policy. Play Store internal testing track.

~1 week
Done when: the AAB passes Play Store internal review and is installable from a tester invite.

Total estimate: ~9–11 weeks part-time. Phases 0–3 ship a credible app on their own; phase 4 lands the headline differentiator; phase 5 turns the job listing's "nice to have" security bullets into shipped features; phase 6 is launch hygiene.

Open questions

Things to decide before they block work.

QuestionBlocksLean
Final app name & package id. Recipe Calculator is the working title. Phase 06 (store listing) Unknown
Should reverse scaling support multiple simultaneous limits ("I have 350 g chicken AND 2 eggs")? Phase 02 No — single limit only in v1. Multi-limit is a constraint solver.
Default unit system on first launch — locale-detect or prompt the user? Phase 06 Locale-detect with override during onboarding.
Tags vs. categories vs. both? Phase 01 Tags only. Categories collapse to "tag with a special name".
How hard should the Play Integrity response be — block the app, or degrade gracefully on a failed verdict? Phase 05 Degrade, don't block. A portfolio app shouldn't lock out a real user over a flaky attestation; log it and soft-warn.
Where does the density table come from? Curate manually, or seed from USDA FoodData Central? Phase 03 Curate ~150 manually. USDA is exhaustive but messy.

Risks

What can sink this and what to do about it.

RiskSeverityMitigation
Ingredient parser quality on real-world photos is too low — users abandon the OCR flow. Warning Confirmation screen sets expectations. Track parse-success metric internally. Iterate the regex set against a captured corpus of real ingredient lists.
Density table coverage gaps produce visibly wrong unit conversions. Warning Fall back to volume-only conversion for unknown ingredients. Show a small "approximate" indicator. Add an "improve this" tap target that logs the miss locally.
Smart rounding rules surprise users on edge cases ("why did my 1.5 eggs become 2?"). Info Always show an adjusted marker beside rounded values. Tap reveals the exact unrounded number.
Jetpack Compose learning curve slows the early phases more than estimated. Info Phase 00 is deliberately a Compose warm-up (theme, nav shell, dev screen) before any hard feature work. The curve is also the portfolio point — budget for it, don't fight it.
Play Integrity / anti-tampering work is fiddly and device-dependent. Info Treat attestation as soft-fail (see open questions). Test on a physical device early in phase 05; keep the security surface small and well-scoped.
Play Store rejection on first submission (icon, privacy policy, permissions justification). Info Submit to internal testing track early in phase 05, not at the end.
Scope creep — meal planning, sharing, sync get added to v1. Critical The non-goals section is the answer. Re-read it when tempted.

Portfolio angle

What this project demonstrates if you point a recruiter at the repo.

  • Modern native Android. Kotlin + Jetpack Compose + Coroutines/Flow + Hilt + Room — the current standard stack, mapped directly onto the target listing.
  • On-device ML + camera. ML Kit text recognition over a CameraX capture flow, running locally — hits two "nice to have" bullets at once, no server, no privacy story to defend.
  • Mobile security. Biometric lock, Keystore-backed encryption, and Play Integrity attestation — the "secure storage, integrity, anti-tampering" line, shipped.
  • Domain modeling. Rational quantities, ingredient-aware unit conversion, smart rounding — non-trivial logic that's easy to evaluate from the code.
  • Shipped, not just coded. A live Play Store listing (even internal testing) ends every "have you shipped?" conversation in one sentence.
