Gradle product flavors

How the play / portfolio split in this app actually works under the hood — written so you can recognize the pattern next time you see it.

The two-axis idea

Android's Gradle plugin lets you build the same app multiple ways at once. The two axes are build types and product flavors, and a variant is the cross-product of one from each axis.

debug build typerelease build type
play flavor playDebug playRelease
portfolio flavor portfolioDebug portfolioRelease

Every Gradle task you see — assemble, test, install — generates a version per variant. assemblePlayDebug builds only the play/debug APK; assemble with no suffix builds all four.

Mental model

Build types are about how you build (debug vs minified release). Flavors are about what you build (which feature set, which audience, which backend). Same app, different shapes.

Source set merging — the part that does the magic

Inside app/src/ there's a directory per source set. When Gradle builds a variant, it merges several source sets into one classpath:

For a portfolioDebug build, three source sets merge into one classpath. playDebug would swap the middle box for src/play/.

If a file with the same fully-qualified name exists in src/main/ and src/portfolio/, that's an error — they'd clash. But if a file only exists in src/portfolio/, it's included only when building portfolio variants. Same for src/play/.

This is the trick that lets SecurityModule.kt exist in both src/play/ and src/portfolio/ with the same class name and package — only one is on the classpath at a time, so there's nothing to conflict.

How the security seam is wired

Three pieces, in this app:

src/main/.../security/AppLockController.ktkotlin
interface AppLockController {
    fun isAvailable(): Boolean
    suspend fun requestUnlock(activity: FragmentActivity): UnlockResult
    // …
}
src/play/.../security/SecurityModule.ktkotlin
@Module @InstallIn(SingletonComponent::class)
abstract class SecurityModule {
    @Binds abstract fun bindAppLockController(
        impl: NoOpAppLockController
    ): AppLockController
    // …
}
src/portfolio/.../security/SecurityModule.ktkotlin
// Same package, same class name, same @Module annotation — different source set.
@Module @InstallIn(SingletonComponent::class)
abstract class SecurityModule {
    @Binds abstract fun bindAppLockController(
        impl: BiometricAppLockController
    ): AppLockController
    // …
}

UI code that needs to lock the app injects AppLockController and never knows which impl Hilt handed it. In the play build, isAvailable() returns false and the Settings row hides. In the portfolio build (once Phase 4 lands), it returns whatever BiometricManager says.

Why this is better than if (BuildConfig.FLAVOR == "play")

Flavor checks scattered through UI code are how this kind of split rots. The interface seam keeps the conditional in exactly one place per feature — the DI binding — and lets the rest of the code stay flavor-agnostic. As a bonus, the play APK doesn't even contain the biometric classes once Phase 4 swaps real impls in; the unused source set isn't compiled at all.

The Gradle config that turns it on

app/build.gradle.ktskotlin
android {
    // …
    flavorDimensions += "audience"
    productFlavors {
        create("play") {
            dimension = "audience"
            // keeps the default applicationId — the Play Store identity
        }
        create("portfolio") {
            dimension = "audience"
            applicationIdSuffix = ".portfolio"
            versionNameSuffix = "-portfolio"
        }
    }
}

Two things to understand:

  • Flavor dimensions let you have multiple orthogonal axes (e.g. audience × backend = 4 flavors total). This app uses one dimension with two values, so the math is trivial.
  • applicationIdSuffix changes the on-device package name so both variants install side-by-side. Without it, installing one would replace the other.

Flavor-specific dependencies

This is the syntax you'll use in Phase 4 when adding the biometric and integrity libraries to portfolio only:

app/build.gradle.ktskotlin
dependencies {
    // shared by both flavors
    implementation(libs.androidx.core.ktx)

    // portfolio only — quoted because there's no compile-time helper for flavor configs
    "portfolioImplementation"(libs.androidx.biometric)
    "portfolioImplementation"(libs.androidx.security.crypto)
    "portfolioImplementation"(libs.play.integrity)
}

The pattern is <flavorName>Implementation(...). AGP generates these configurations for you per flavor; the string-quoted form is just to satisfy the Kotlin DSL's type system.

Commands you'll actually run

CommandWhat it does
./gradlew :app:assemblePlayDebug Build only the play/debug APK
./gradlew :app:assemblePortfolioDebug Build only the portfolio/debug APK
./gradlew :app:installPortfolioDebug Build + install portfolio on the connected device
./gradlew :app:testPlayDebugUnitTest Run unit tests for the play flavor
./gradlew :app:assemble Build all four variants
Android Studio UI

The Build Variants panel (View → Tool Windows → Build Variants) lets you switch which variant Studio uses for "Run". When you flip to portfolioDebug, Studio compiles only that variant and your IDE sees the portfolio source set as part of the project — files in src/play/ are greyed out. This trips a lot of people up: "why doesn't my code see this class?" is usually because the wrong variant is selected.

Side-by-side install proof

Because the portfolio flavor declares applicationIdSuffix = ".portfolio", the two APKs have different package identifiers and Android treats them as different apps:

shellbash
adb shell pm list packages | grep recipecalculator
# package:io.github.chwi.recipecalculator           ← play build
# package:io.github.chwi.recipecalculator.portfolio ← portfolio build

This is the win: on your phone the portfolio build is a separate icon you can open during interviews; the play build is what you actually use.

Where to go from here

  • src/play/.../core/security/*.kt — three no-op impls + SecurityModule

Phase 4 work doesn't touch the seam — it just swaps the portfolio bodies for real implementations, adds portfolioImplementation dependencies, and updates the portfolio AndroidManifest.xml with the biometric permission. The play side never changes.
