Phase 4 — Security & Hardening

Four loosely-related Android security primitives, why they exist, and how each one fits a local-only recipe app. Roughly two days of work, broken into four cleanly-committable pieces.

TL;DR

You're shipping four things: a biometric prompt in front of the app (or specific recipes), Keystore-backed encryption for a small set of sensitive files/prefs, a Play Integrity check on launch that warns but doesn't brick, and an XML backup-rules file that tells Google's auto-backup what to sync to Drive and what to leave alone. None of these are individually hard — the value is in knowing why you'd choose each, what the threat model is, and where the gotchas live.

The reason these four are bundled into one phase isn't that they're technically related. It's that they all show up on Android job listings under the same heading ("secure storage / biometrics / anti-tampering"), and shipping all four with intent — not just copied from a Codelab — is what separates a portfolio that gets a callback from one that doesn't.

What you're actually protecting

Each piece defends against a different threat. Knowing which is which is the part most tutorials skip. The recipe app is local-only, so the threat model is narrower than a banking app's — but each control still has a clean answer.

ControlDefends againstConcretely, what stops working for the attacker
Biometric lock Someone picks up your unlocked phone and opens the app They can't get past the prompt without your fingerprint / face / device PIN.
Keystore encryption Someone with physical access dumps app files (rooted device, ADB backup, file manager) The bytes on disk are ciphertext. The key lives in tamper-resistant hardware and can't be exfiltrated even with root.
Play Integrity Someone repackages your APK, runs it on an emulator, or sideloads a tampered build The verdict comes back as UNRECOGNIZED_VERSION / not MEETS_DEVICE_INTEGRITY and your app can refuse to operate (or degrade gracefully).
Backup rules Google's auto-backup syncs sensitive data to the user's Drive without your intent The excluded files never leave the device. Includes go to Drive; excludes don't.

Notice that none of these defend against a remote network attacker — this is a local app with no backend. That's a feature, not a gap. When you write this up in the README, name the threats you're addressing and name the ones you're not.

1 · Biometric app lock half-day

The user opts in via a Settings toggle. When the app comes to the foreground (cold launch, or warm resume after N minutes in the background), a system biometric prompt appears in front of the UI. If they pass it, they see the app. If they cancel, they see a "Unlock to continue" screen with a retry button.

What you'll use

  • androidx.biometric:biometric:1.2.0-alpha05 — the Jetpack wrapper around the platform BiometricPrompt. (1.1.0 is the latest stable; the alpha line has Compose-friendlier APIs.)
  • androidx.biometric.BiometricPrompt — the prompt itself, attached to your FragmentActivity.
  • androidx.biometric.BiometricManager — query availability before showing the prompt.

The minimum viable wiring

AppLock.ktkotlin
fun showBiometricPrompt(
    activity: FragmentActivity,
    onSuccess: () -> Unit,
    onCancel: () -> Unit,
) {
    val executor = ContextCompat.getMainExecutor(activity)
    val prompt = BiometricPrompt(activity, executor,
        object : BiometricPrompt.AuthenticationCallback() {
            override fun onAuthenticationSucceeded(result: …) = onSuccess()
            override fun onAuthenticationError(code: Int, msg: CharSequence) = onCancel()
        })

    val info = BiometricPrompt.PromptInfo.Builder()
        .setTitle("Unlock Recipe Calculator")
        .setAllowedAuthenticators(BIOMETRIC_STRONG or DEVICE_CREDENTIAL)
        .build()
    prompt.authenticate(info)
}

In this app, that's three pieces

  1. A Settings toggle ("Lock app with biometrics") — a new boolean in your existing SettingsRepository.
  2. A lifecycle observer in MainActivity (or a top-level Composable) that tracks ON_STOP / ON_START, computes time-in-background, and re-locks if > N minutes. androidx.lifecycle's ProcessLifecycleOwner is the right hook.
  3. A locked overlay Composable that sits above the nav host while isLocked is true. Tapping it triggers showBiometricPrompt.

Optionally, a per-recipe isLocked: Boolean column (Room migration v2) lets the user mark individual recipes private — the same prompt fires when they tap such a recipe. That's a nice differentiator and shows you can do a Room migration.

Gotcha

Always call BiometricManager.from(ctx).canAuthenticate(BIOMETRIC_STRONG or DEVICE_CREDENTIAL) before showing the prompt and branch on the result. On a phone with no enrolled fingerprint and no device PIN, the prompt will fail immediately — you want a friendly "set up a screen lock to enable this" message, not a crash dialog.

Gotcha

The emulator doesn't have a biometric by default. In the AVD's extended controls (the ⋯ button on the emulator window) go to Fingerprint and use "Touch Sensor" to simulate a scan. The device PIN fallback works without configuration.

2 · Encryption at rest half-day

You need to know two layers here, and the second is built on the first.

Android Keystore

A platform service that stores cryptographic keys in tamper-resistant hardware (TEE / StrongBox on modern devices). You ask it to encrypt/decrypt; the key never leaves the secure hardware. Even adb root can't pull it.

Jetpack Security

A wrapper that gives you two friendly APIs on top of Keystore: EncryptedSharedPreferences (encrypted key-value prefs) and EncryptedFile (encrypted file I/O). The keys it uses live in Keystore.

You won't see Keystore directly in your code — you'll use Jetpack Security and trust it to wire Keystore underneath.

What to encrypt

The honest answer: for a recipe app, very little is actually sensitive. The portfolio answer: pick a clean, narrow surface so you show the API without over-engineering.

  • The biometric-lock setting (a boolean in EncryptedSharedPreferences) — small, isolated, demonstrates the prefs API.
  • An optional "private notes" field per recipe, stored as an EncryptedFile in filesDir/private_notes/<recipe-id>.enc — demonstrates the file API and gives "locked" recipes a meaningful payload to protect.

Dependency and use

app/build.gradle.ktskotlin
// stable; older but well-documented
implementation("androidx.security:security-crypto:1.0.0")
SecurePrefs.ktkotlin
val masterKey = MasterKey.Builder(context)
    .setKeyScheme(MasterKey.KeyScheme.AES256_GCM)
    .build()

val prefs = EncryptedSharedPreferences.create(
    context,
    "secure_prefs",
    masterKey,
    EncryptedSharedPreferences.PrefKeyEncryptionScheme.AES256_SIV,
    EncryptedSharedPreferences.PrefValueEncryptionScheme.AES256_GCM,
)
Heads-up on library state

androidx.security:security-crypto:1.0.0 is the stable release everyone uses. The 1.1.0-alpha line has been alpha for a long time and Google has signaled the library is in maintenance. For a portfolio that's fine — use 1.0.0, note in your README that you're aware of the situation, and that for a greenfield production project you'd evaluate Tink directly. Showing that you read past the first page of docs is the point.

Where it plugs in

  • Hilt module providing a @Singleton SecurePreferenceStore wrapping EncryptedSharedPreferences.
  • Settings repo writes the biometric-lock flag here instead of plain SharedPreferences / DataStore.
  • The "private notes" feature, if you build it: a small PrivateNotesRepository that takes a recipe ID and uses EncryptedFile for read/write.
Why not encrypt the whole Room database?

You can — via SQLCipher integration (net.zetetic:android-database-sqlcipher + SupportFactory with a passphrase). It's a heavier lift and the wrapper has its own opinions. For a portfolio piece on a recipe app, encrypting an opt-in private-notes file is more proportionate to the threat model and easier to explain in an interview ("I encrypt the data the user has marked private, not their entire library").

3 · Play Integrity API half to full day

Google's anti-tampering / anti-cheat service. It replaced the older SafetyNet Attestation API in 2024. Conceptually, it lets your app ask Google "am I really running where I claim to be running, on a real device, as a non-tampered build?" — and get back a signed token with the verdict.

The token comes back signed. You either verify the signature server-side (typical) or parse the verdict locally as best-effort (acceptable for a local-only app — declare this trade-off honestly).

What the verdict actually contains

  • appRecognitionVerdict — PLAY_RECOGNIZED (the build matches a Play release), UNRECOGNIZED_VERSION (modified APK, sideloaded), UNEVALUATED (couldn't check).
  • deviceRecognitionVerdict — set of labels: MEETS_DEVICE_INTEGRITY, MEETS_BASIC_INTEGRITY, MEETS_STRONG_INTEGRITY. A rooted phone fails the first two.
  • appLicensingVerdict — LICENSED if the user got the app from Play, UNLICENSED otherwise.

For this app: soft-fail

The architecturally honest move for a local-only app is soft-fail: call Integrity once on launch, log/store the verdict, and if it's bad show a non-blocking warning ("This device looks tampered — some features may be unavailable"). Don't refuse to run. Reasons:

  • No backend means you can't verify the token signature server-side, so you're trusting the local response — bricking on a partially-trusted check is bad UX.
  • False positives on legitimate devices (custom ROMs, work profiles) are real and annoying.
  • Quota limits matter — 5,000 standard requests/day per Play project on the free tier.

Wiring

app/build.gradle.ktskotlin
implementation("com.google.android.play:integrity:1.4.0")
IntegrityCheck.ktkotlin
suspend fun checkIntegrity(context: Context): IntegrityVerdict {
    val manager = IntegrityManagerFactory.create(context)
    val nonce = UUID.randomUUID().toString()
    val token = manager
        .requestIntegrityToken(IntegrityTokenRequest.builder().setNonce(nonce).build())
        .await()
        .token()
    return parseVerdict(token)
}
Setup, not just code

Integrity requires registration in Play Console under App integrity → Integrity API. You'll get a project number and need to opt the app in. Without that step the API returns errors. You can use the standard requests path with a real Play Console (internal test track is enough) without needing a backend.

Gotcha

You won't get meaningful verdicts from an Android Studio "run" install — it's not Play-installed and not signed with the upload key. Test on an internal-track install from Play Console. Plan a half-day for the Play Console setup the first time you do it.

4 · Scoped backup rules 1–2 hours

Android Auto Backup will, by default, sync your app's data directory to the user's Google Drive (up to 25MB per app). That's mostly good — when they reinstall on a new phone they get their recipes back. But you want explicit control over what's included and what isn't. Backup rules are how you do that.

Two files, two scenarios

FileWhen it appliesWhat it controls
backup_rules.xml Android 11 and below Auto-backup to Drive (the classic flow).
data_extraction_rules.xml Android 12+ Separate rule sets for cloud-backup (Drive) and device-transfer (new "switch phones" flow).

Both files go in app/src/main/res/xml/. Wire them up in AndroidManifest.xml:

AndroidManifest.xmlxml
<application
    android:fullBackupContent="@xml/backup_rules"
    android:dataExtractionRules="@xml/data_extraction_rules"
    …>

Reasonable rules for this app

res/xml/data_extraction_rules.xmlxml
<?xml version="1.0" encoding="utf-8"?>
<data-extraction-rules>
    <cloud-backup>
        <!-- INCLUDE: user content the user wants on a new phone -->
        <include domain="database" path="recipe_db"/>
        <include domain="file" path="recipe_photos/"/>
        <include domain="sharedpref" path="display_settings.xml"/>

        <!-- EXCLUDE: keystore-bound prefs (key handles don't migrate), transient OCR temp dir -->
        <exclude domain="sharedpref" path="secure_prefs"/>
        <exclude domain="file" path="ocr_captures/"/>
        <exclude domain="file" path="private_notes/"/>
    </cloud-backup>
    <device-transfer>
        <!-- device-transfer can include more, since both endpoints are trusted -->
        <include domain="database" path="recipe_db"/>
        <include domain="file" path="recipe_photos/"/>
        <exclude domain="file" path="ocr_captures/"/>
    </device-transfer>
</data-extraction-rules>

Why each choice:

  • Include the Room DB — that's the actual recipe library. The reason this feature exists.
  • Include photo files — recipes without photos on the new phone are sad.
  • Exclude secure_prefs — it's encrypted with a Keystore key that's bound to this device. The ciphertext can't be decrypted on a different device anyway, so syncing it is just confusing.
  • Exclude ocr_captures/ — transient working files. Pure noise in a backup.
  • Exclude private_notes/ for cloud backup, same Keystore-binding reason. (You could include them in device-transfer, since modern transfers use a more secure channel, but for v0.5.0 just exclude both.)
Gotcha

Test backup by hand — don't trust that "it'll work in prod." Trigger a manual backup from adb:

shellbash
adb shell bmgr backupnow io.github.chwi.recipecalculator
adb shell bmgr list transports
adb shell bmgr restore <token> io.github.chwi.recipecalculator

The exclusions don't apply retroactively to backups already in Drive, so test before you publish.

Suggested build order

Smallest piece first, biggest piece last. Each step is a clean separate commit.

  1. Backup rules ~1–2h

    Pure XML. Gets you into the security headspace. No new dependency. Test with bmgr backupnow.

  2. Biometric app lock ~half day

    Most visible piece — easy to demo in a screen recording for the portfolio. Add the Settings toggle, the lifecycle gate, the locked overlay. Optionally the per-recipe lock + Room migration v2.

  3. Encryption (EncryptedSharedPreferences) ~half day

    Move the biometric-lock flag (and any other small sensitive prefs) into secure_prefs. Wire a Hilt provider. Optional: the private-notes file feature as a second example of the API.

  4. Play Integrity (soft-fail) ~half to full day

    Last because it requires Play Console registration. Add a one-shot Integrity check on launch, store the verdict, surface a warning banner if it's bad. Document the soft-fail choice in the README.

If you only have one day, do steps 1 + 2 and ship as v0.4.1. Steps 3 + 4 can land as v0.5.0 a week later. Don't bundle.

Portfolio framing

The most common mistake on portfolio security work is shipping all four features but writing them up as "I added X, Y, Z." That reads like a Codelab. The version that reads like a senior engineer:

README pattern that works

For each control, name the threat it addresses, the scope you chose (and what you deliberately didn't do), and any known limitation. E.g. for Integrity: "Soft-fail by design — the app is local-only and there is no backend to verify the token signature against Google's public keys. A production deployment with sensitive operations would verify server-side and refuse to operate on a bad verdict." That single sentence demonstrates more judgement than the code change does.

Three things to call out in an interview, in your own words:

  • Why BIOMETRIC_STRONG or DEVICE_CREDENTIAL — covers users without enrolled biometrics, gives a clean PIN/pattern fallback, satisfies the "strong" classification for Keystore key auth-bound usage if you ever extend that way.
  • Why EncryptedSharedPreferences for a narrow surface, not whole-DB SQLCipher — proportionality to threat model, lower dep weight, easier to reason about.
  • Why soft-fail Integrity — no backend to verify against, false positives on legit devices, UX of bricking on a partially-trusted check is worse than degrading gracefully.

Each of these is a small judgement call. Stringing the three of them together is what makes the work look senior.
